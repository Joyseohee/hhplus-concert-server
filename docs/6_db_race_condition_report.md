## 1. 배경 (Background)

* **시스템 개요**

    * 콘서트 예약 시스템: 토큰 발급 → 좌석 점유 → 예약 확정 → 결제/충전 흐름
* **운영 환경 및 가정**

    * MySQL 8.x, InnoDB, 격리 수준: Repeatable Read
    * 스케줄러 주기: 1분
    * 최대 동시 요청: 100 TPS(가정)
* **목적**

    * 주요 동시성 시나리오 식별
    * MySQL 기본 메커니즘만으로 안전·고성능 보장

## 2. 문제 (Problem)

1. **동일 좌석 동시 점유**
2. **점유 만료 후 예약 경쟁**
3. **토큰 활성화 경쟁**
4. **토큰 만료와 사용 경쟁**
5. **잔액 충전 동시 업데이트**
6. **잔액 차감 (예약 확정 시)**
7. **충전 ↔ 차감 혼합 요청**

* **영향**: 중복 점유, 오버차지, 순서 뒤바뀜, 퍼포먼스 저하, 데드락 등

## 3. 해결방법 (Solution)

> **MySQL InnoDB Repeatable Read** 환경에서
>
> * **필요할 때만** 비관적 락으로 순서 보장
> * **나머지는** 낙관적 락+재시도로 최소 블로킹
> * **DB 제약·쿼리 필터링**으로 단순·효율적 제어

1. **동일 좌석 동시 점유**

    * **해결**: `UNIQUE(concert_id, seat_id)` 제약
    * **이유**: DB가 원자적으로 중복 차단 → 별도 락·트랜잭션 불필요

2. **점유 만료 후 예약 경쟁**

    * **해결**:

        * 로직 내 “만료 여부 확인 → `IllegalArgumentException`”
        * `@Retryable(ObjectOptimisticLockingFailureException)` 재시도
    * **이유**: 블로킹 없이 충돌 해결, 일관성 보장

3. **토큰 활성화 경쟁**

    * **해결**: `SELECT … FOR UPDATE` / `@Lock(PESSIMISTIC_WRITE)`
    * **이유**: 로우 블로킹으로 순차 처리 보장, race-free

4. **토큰 만료와 사용 경쟁**

    * **해결**: 검증 시 `WHERE status='ACTIVE' AND expiresAt>NOW()`
    * **이유**: 단일 쿼리로 안전 체크, 락 불필요

5. **잔액 충전 동시 업데이트**

    * **해결**: `@Version` (낙관적 락) + `@Retryable(...)` 재시도
    * **이유**: 짧은 재시도로 높은 처리량

6. **잔액 차감 (예약 확정 시)**

    * **해결**:

        * `@Version` 기반 락
        * 예약 전 `validateQueueTokenService`로 토큰 1회성 검사
    * **이유**: 중복 차감 근본 차단, 즉시 실패 처리

7. **충전 ↔ 차감 혼합 요청**

    * **해결**: 양쪽 모두 `@Version` + 재시도
    * **이유**: 가환 연산으로 순서 무관, deadlock 위험 극소

## 4. 실험결과 (Experiment Results)

* **환경**: JVM8, MySQL InnoDB, Kotest 멀티스레드 테스트
* **지표**:

    * 성공률(%)
    * 평균 응답시간(ms)
    * 재시도 횟수
    * 데드락 발생 횟수
* **예시 차트**:

    * TPS vs. 스레드 수
    * 오류 유형별 비율

## 5. 한계점 (Limitations)

* 극한 부하(>200 TPS) 시 데드락 가능성
* 스케줄러 지연에 따른 만료 롤링 문제
* 파티셔닝/샤딩 도입 시 UNIQUE 제약 적용 어려움
* 주요하지 않은 시나리오가 하나의 트랜잭션에 묶여 있음
  * 예: 좌석 예약 시 결제-예약과 토큰 만료가 하나의 트랜잭션으로 묶임
  * 추후 이벤트 기반 처리로 분리 필요

## 6. 결론 (Conclusion)

* **요약**: 각 시나리오별 최적 메커니즘 적용으로 안정성·성능 확보
* **추천**:

    * 모니터링: `innodb_lock_wait_timeout` 로그, APM
    * 운영 조치: Canary 배포, 기능 토글
* **향후 과제**:

    * Redis 분산 락 PoC
    * 파티셔닝 환경 테스트
